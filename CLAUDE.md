## 배포 워크플로우 (중요)
1. 코드 수정
2. `npm run build`로 타입/문법 에러 사전 검증 (필수)
3. `git add . && git commit -m "메시지" && git push`
4. Vercel 자동 배포 (CDN 전파 1~2분)

## 자율 작업 워크플로우 (A 방식 — 가장 중요)

주진님은 "방아쇠"만 당긴다. 한 번의 요청을 받으면, 그 안에서 아래 루프를 **스스로 끝까지** 돈다. 추가 크레딧이 드는 백그라운드 폴링은 절대 하지 않는다 (요청이 있을 때만 작동).

### 표준 루프 (요청 1건당)
1. **이해** — 요청을 파악. 애매하면 추측해서 진행하되 가정을 명시. (사소한 건 묻지 말고 진행)
2. **수정** — 코드 작성/수정
3. **빌드 검증** — `npm run build`로 타입/문법 에러 확인. 에러 나면 **스스로 고치고 재빌드** (주진님께 묻지 말 것)
4. **자가 확인** — Claude Preview로 직접 띄워 검증:
   - `preview_start` → `preview_screenshot`
   - ⚠️ 인트로 오버레이가 먼저 뜸 → `preview_click`으로 SKIP(`#oc-skip`) 누른 뒤 실제 화면 확인
   - 관련된 페이지를 실제로 보고, 모바일 뷰도 확인
   - 콘솔 에러/404/깨짐 있으면 **스스로 수정 → 3번으로 돌아가 재검증**
5. **배포** — 객관적으로 정상이면 `git add . && git commit && git push` (Vercel 자동 배포)
6. **보고** — 무엇을 했는지 간결히 요약. 스크린샷 첨부.

### 자율로 처리 (묻지 않고 스스로 끝냄)
- 빌드 에러, 타입 에러, 문법 에러
- 콘솔 에러, 404, 깨진 레이아웃, 안 눌리는 버튼
- 무한 로딩(setLoading 누락), 모바일 글자/세로 깨짐
- 명백한 버그 — "객관적으로 깨진 것"은 전부 스스로 고친다

### 멈추고 주진님께 물어볼 것 (A/B 후보로 제시)
- **미적 판단**: "싸 보인다/비어 보인다/영광스럽지 않다" 류 → 봇이 판단 불가. 후보 2개 만들어 선택 요청.
- **방향 결정**: 기능을 넣을지 말지, 어떤 컨셉으로 갈지
- **되돌릴 수 없는 작업**: DB 스키마 변경(마이그레이션 SQL은 주진님이 실행), 데이터 삭제, 결제 관련
- **전략 판단**: 우선순위, 마케팅, 수익화

### 핵심 원칙
- "한 번의 요청 안에서" 자율로 도는 것이지, 아무도 요청 안 했는데 스스로 일을 찾지 않는다 (크레딧 절약).
- 객관적 품질(버그/빌드/깨짐)은 끝까지 책임지고, 주관적 판단(미적/전략)만 주진님께 넘긴다.
- 배포 전 항상 빌드 + 자가확인. "될 것 같다"로 배포 금지, 실제로 띄워 보고 배포.

### 점검 요청 패턴
주진님이 "X 페이지 점검해줘" 하면: Preview로 띄워 → 콘솔/네트워크/모바일뷰 확인 → 발견한 객관적 문제는 스스로 수정 → 애매한 건 목록으로 보고. (24시간 자동 점검이 아니라 요청 시에만)

## 코딩 원칙 (카파시 4원칙 — 영상 학습 적용)
> 출처: `andrej-karpathy-skills` (★10만+ CLAUDE.md). 핵심: "AI는 코드를 못 짜는 게 아니라 너무 빨리·자신있게 짠다 → 브레이크가 필요하다." 아래 워크플로우와 충돌 시 워크플로우(자율성) 우선.

1. **생각 먼저** — 추측 금지. 해석이 여러 갈래거나 트레이드오프가 있으면 멋대로 정하지 말고 선택지를 제시(더 단순한 대안 포함). 단, 사소·객관적인 건 워크플로우대로 묻지 말고 진행 — "진짜 갈리는 설계 결정"에서만 A/B로 멈춘다.
2. **단순함 우선** — 문제 푸는 최소 코드만. 요청 안 한 기능/추상화/유연성 추가 금지. 200줄을 50줄로 줄일 수 있으면 다시 짠다. 시니어가 "이거 복잡한데" 할 것 같으면 단순화.
3. **수술하듯 수정** — 바뀐 모든 줄이 요청과 직접 연결돼야 한다. 아니면 건드리지 않는다. 옆 코드·주석·서식 멋대로 개선/리팩터 금지(기존 스타일 유지). 관련 없는 죽은 코드를 발견하면 지우지 말고 **보고**. (형태가 다른 중복은 무리해서 통합하지 말 것 — 결합도만 늘어남.)
4. **목표 기반 실행** — "버그 고쳐"가 아니라 "버그 재현 테스트 → 통과"처럼 **검증 가능한 목표**로 바꿔 실행. 이미 이 프로젝트의 `build` + Preview 자가검증 루프가 이 원칙의 구현체 → 계속 유지·강화.

## 흔한 함정 (학습됨)
- **setLoading(false) 누락** → 무한 LOADING. 모든 fetch useEffect에서 확인.
- **쿠키로 인증 체크 금지**: `document.cookie.includes("sb-")` 부정확. 페이지 자체 `getUser()`에 위임.
- **PostgREST 조인은 FK 필요**: `profiles(nickname)` 조인이 FK 없으면 조용히 빈 배열 → 개별 조회로 대체.
- **RLS 누락 = 조용한 실패**: insert가 막혀도 에러 없이 넘어감. 모든 쓰기에 error 핸들링.
- **.jsx에 TypeScript 문법 금지**: `(c: any)` 쓰면 Turbopack 파싱 에러 → .tsx로 바꾸거나 타입 제거.
- RLS 정책은 Supabase SQL 에디터에서 수동 추가 (Claude가 SQL 제공, 주진님 실행).

## 현재 단계 & 방향
- **출시 준비 완료 → 마케팅 단계 진입** (2026-06 기준). 출시 차단 요소(개인정보방침 PIPA·가입 동의·문의 API 레이트리밋·신고/가이드라인) 모두 해소. 출시 준비도 ≈ **70/100**.
